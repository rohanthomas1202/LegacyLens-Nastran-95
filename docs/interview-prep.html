Elevator Pitch

CODEX-95 is a RAG-powered code intelligence platform that makes NASA's NASTRAN-95 Fortran 77 codebase queryable through natural language. Instead of manually searching through 418,000+ lines of legacy code across 1,846 files, developers can ask questions like "Where is the main entry point?" or "How does stiffness matrix assembly work?" and get streaming AI-generated answers with specific file and line references. It also features an interactive dependency graph, code modernization (Fortran→Python/C/Java/Rust), code flow tracing, dead code detection, and more.

One-liner: "I built an AI-powered code intelligence platform that uses RAG to let developers query, visualize, and modernize a 418K-line NASA Fortran codebase — with streaming AI answers, an interactive 2D/3D dependency graph, and Fortran-to-modern-language translation."

Ingestion Pipeline

How code gets from files to vectors

File Discovery → Preprocessor → Chunker → Embedder → Pinecone

Step 1: File Discovery

Recursively scans codebases/nastran95/ for supported file extensions (.f, .f77, .f90, .c, .h). Skips bin/, inp/, um/ directories. Detects language from extension.

Result: 1,846 files, 418,349 lines of code

Step 2: Preprocessing

  • Encoding detection with chardet (legacy files may not be UTF-8)
  • Fortran 77: truncate columns 73-80 (sequence numbers, not code)
  • Normalize line endings, strip trailing whitespace, collapse blank lines

Step 3: Chunking (the core module)

Syntax-aware parsing — see Chunking section

Step 4: Embedding

Batches of 100 chunks → OpenAI API. Prepends metadata header before embedding. See Embedding section

Step 5: Vector Storage

Upserts to Pinecone in batches of 100 with metadata (file_path, lines, type, name, language, content, dependencies)

Retrieval Pipeline

From query to answer

User Query → Embed → Pinecone Search (top-5) → Rerank → Claude Generation → Answer

Semantic Search

The query is embedded with the same OpenAI model used for ingestion. Pinecone returns top-k results (default 5) by cosine similarity. Supports optional filters by language and file path.

Fortran 77 Chunking (Core Innovation)

Why this is the hardest part of the project

Fortran 77 Fixed-Form Format

Columns:  1     6   7                    72  73      80
          |     |   |                     |  |        |
          Label Cont Source Code              Seq Nums

C     THIS IS A COMMENT (C in column 1)
      SUBROUTINE TEST(A,B,C)         (code in cols 7-72)
      INTEGER A,B,C
     +  CONTINUED_LINE               (+ in col 6 = continuation)
      A = B + C
      RETURN
      END                            12345678 (cols 73-80 stripped)

Boundary Detection Rules

  1. Scan columns 7+ for keywords: SUBROUTINE, FUNCTION, PROGRAM, BLOCK DATA, ENTRY
  2. Track bare END statements as routine terminators
  3. Group continuation lines (non-blank/non-zero in column 6) with parent
  4. Skip comments (C/*/! in column 1) for boundary detection
  5. Extract dependencies: CALL targets, COMMON blocks, INCLUDE files

Chunk Data Model

@dataclass
class Chunk:
    content: str         # Full source code
    file_path: str       # Relative path (e.g., "bd/sdr2a.f")
    start_line: int      # First line (1-based)
    end_line: int        # Last line
    chunk_type: str      # subroutine|function|program|block-data|entry|fixed-size
    name: str            # Routine name (e.g., "SDR2A")
    language: str        # fortran|c
    dependencies: list   # ["CALL:TARGET", "COMMON:BLOCK", "INCLUDE:FILE"]

Why Not AST-Based?

No production-quality Fortran 77 parser exists for Python. The fixed-form format makes regex-based parsing reliable — column positions are deterministic, not context-dependent. Our approach correctly identifies 100% of SUBROUTINE/FUNCTION boundaries.

Fallback

Files without detectable routine boundaries use fixed-size chunking: 40 lines with 10-line overlap.

Before embedding, we prepend a structured header to give the model context:

This helps the embedding model understand the code's role and location.

Rate Limit Handling

  • Batch size: 100 chunks per API call
  • 0.5 second delay between batches
  • Exponential backoff on rate limit errors (2^attempt + 1 seconds, up to 5 retries)
  • Token truncation: chunks over 8,000 tokens get lines removed iteratively (20% at a time)

Graph & Visualization

The graph builder (graph_builder.py) assembles these into a CallGraph with nodes, edges, reverse edges, and COMMON block mappings. The graph is cached to graph_cache.json for fast subsequent loads.

Code Flow Tracer

BFS shortest path between any two routines. The UI shows the path as a step-by-step sequence with file locations for each hop. Useful for understanding execution flow through the codebase.

Dead Code Detection

Traverses from known entry points (PROGRAM blocks) through the call graph. Any routine with zero incoming edges that isn't an entry point is flagged as potentially dead. Reports stats (total routines, reachable count, dead count, coverage percentage) and lists all dead routines with file locations.

Key Technical Decisions

Why I chose what I chose

Why syntax-aware chunking instead of fixed-size?
Fixed-size chunks break mid-function, losing context. Syntax-aware chunks preserve complete subroutines as units, so the embedding captures the full semantic meaning. This dramatically improves retrieval precision — the system returns the exact routine you're asking about, not a fragment.
Why keyword reranking instead of a cross-encoder?
Cross-encoders (like ms-marco-MiniLM) add 200-500ms latency and infrastructure complexity. For code search, exact name matching is the highest-value signal — users often know the routine name. A simple +0.1 boost for name matches solves the most common failure mode (exact match ranked below semantic match) at zero additional cost.
Why OpenAI for embeddings + Claude for generation?
Vendor diversity: if one provider has an outage, only half the system is affected. Also leverages each provider's strengths — OpenAI has battle-tested embeddings, Claude has excellent code understanding and instruction following.
Why Pinecone over ChromaDB or pgvector?
Zero infrastructure management. Pinecone serverless auto-scales, has a generous free tier (100K vectors), and supports native metadata filtering. ChromaDB would require self-hosting, pgvector needs a Postgres instance. For a demo project with 4K vectors, the free tier is perfect.
Why lazy-load the 3D graph instead of including it by default?
Three.js and react-force-graph-3d add ~1.3MB to the bundle. Most users only need the 2D view. By using React.lazy() and dynamic import(), the 3D code is only downloaded when the user toggles to 3D mode. This keeps the initial bundle small and the app fast to load.
Why component-based architecture instead of a single file?
The app grew to include 9 features, 4 tabs, modals, graph visualizations, autocomplete, and streaming. At that scale, a single file becomes unmaintainable. The 12-component architecture keeps each file focused (QueryTab handles queries, DependencyGraph handles the graph, etc.) and enables lazy loading for expensive components like the 3D graph.

Challenges & Solutions

Problems I hit and how I solved them

OpenAI Rate Limits During Ingestion
Problem: Embedding 4,040 chunks in rapid succession hit the 1M token/minute rate limit.
Solution: Exponential backoff (2^attempt + 1 seconds, up to 5 retries) plus 0.5s delay between batches. The ingestion completes in ~4 minutes without manual intervention.
Token Limit Exceeded on Large Routines
Problem: Some NASTRAN routines exceed 1,000 lines, producing chunks with 15,000+ tokens — far over the 8,191 token embedding limit.
Solution: Iterative truncation: if a chunk exceeds 8,000 tokens, remove 20% of lines from the end and re-count. Repeat until under limit. This preserves the function signature and early logic.
Windows Backslash Paths
Problem: Ingestion on Windows stored file paths with backslashes (e.g., mis\sub.f). On Linux (Railway), the file lookup failed because files are at mis/sub.f.
Solution: Normalize backslashes to forward slashes in the /api/file endpoint, and use .as_posix() in file discovery for future ingestions.
Railway Docker PORT Variable
Problem: Railway injects $PORT as an environment variable, but the Dockerfile CMD wasn't expanding it properly.
Solution: Used Python to read os.environ.get('PORT', '8000') and pass it directly to uvicorn, bypassing shell expansion issues.
Encoding Detection for Legacy Files
Problem: Some 1970s-era Fortran files use non-UTF-8 encodings.
Solution: chardet library detects encoding automatically, with UTF-8 fallback using errors='replace'.

Common Interview Questions

Q: What is RAG and why did you use it?
RAG (Retrieval-Augmented Generation) combines information retrieval with LLM generation. Instead of asking an LLM to answer from its training data (which may be outdated or hallucinate), we first retrieve relevant documents from a vector database, then augment the LLM's prompt with those documents, and the LLM generates an answer grounded in actual source code. This ensures answers cite real files and line numbers.
Q: How do you handle hallucinations?
Three strategies: (1) The system prompt requires Claude to cite specific file:line references for every claim. (2) We include the actual source code in the context, so the LLM can only reference code that exists. (3) The user sees the source chunks alongside the answer, so they can verify claims against the actual code.
Q: What happens when the query doesn't match any code?
Pinecone always returns results (cosine similarity finds the closest matches), but the scores will be low. The reranker won't boost irrelevant results. Claude receives low-relevance chunks and its system prompt instructs it to note when it's uncertain or when the retrieved code may not be directly relevant.
Q: How would you scale this to a larger codebase?
The architecture scales naturally: Pinecone serverless handles millions of vectors, embeddings are batched, and the chunker is per-file. For a 10x larger codebase, ingestion takes ~40 minutes instead of 4, and costs ~$1 instead of $0.10. The retrieval pipeline is unchanged — Pinecone search is O(log n).
Q: Why Fortran 77 specifically?
NASTRAN-95 is a real NASA codebase used in structural analysis. Fortran 77's fixed-form format (column-based, continuation markers, comment conventions) makes it a challenging but tractable chunking problem. The fixed-form format is actually more predictable than free-form languages, making regex-based parsing reliable.
Q: What would you improve if you had more time?

I've already shipped several features that were originally "stretch goals" — streaming responses, the dependency graph, and code modernization. What's still on the roadmap:

  • Conversational mode: Multi-turn chat with codebase context so follow-up questions build on previous answers
  • Semantic code diff: Compare two routines side-by-side and highlight structural (not just textual) differences
  • Query caching: LRU cache for frequent queries (30-50% cost savings)
  • Evaluation framework: Automated precision@5 testing with labeled queries
  • Self-hosted embeddings: Replace OpenAI embeddings with sentence-transformers for 100% embedding cost elimination
Q: Walk me through what happens when a user submits a query.
  1. Frontend sends POST to /api/query/stream with the query text
  2. Backend embeds the query using OpenAI text-embedding-3-small (same model as ingestion)
  3. Embedded query goes to Pinecone for cosine similarity search, returns top-5 chunks
  4. Reranker boosts scores: +0.1 for name matches, +0.02 for content keyword matches
  5. Backend sends an SSE sources event with the ranked source chunks — the frontend immediately renders the source cards
  6. Top chunks are sent to Claude Sonnet with a NASTRAN-specific system prompt, using the streaming API
  7. Each token is streamed back as an SSE token event — the answer appears in real-time
  8. A final done event carries token usage stats
  9. Frontend renders: streaming answer with markdown formatting, source cards with syntax highlighting, View File buttons
Time to first token: ~1 second (embedding + Pinecone + Claude start). Full answer: ~2-3 seconds.

Deep Dive Technical Q&A

For senior/staff-level interviews

Q: Why cosine similarity over dot product or Euclidean distance?
Cosine similarity measures the angle between vectors, not magnitude. This matters because OpenAI embeddings are normalized — two chunks about similar topics will point in similar directions regardless of text length. Dot product would favor longer texts (higher magnitude). Euclidean distance works but is less intuitive for similarity ranking.
Q: How do you handle the 8,191 token limit for embeddings?
We set a safety margin at 8,000 tokens. If a chunk (with metadata header) exceeds this, we iteratively remove 20% of lines from the end and re-count tokens using tiktoken (cl100k_base encoding). This preserves the function signature and early logic, which are most semantically important. In practice, only ~2% of NASTRAN routines exceed the limit.
Q: What's the difference between your ingestion embedding and query embedding?
They use the same model (text-embedding-3-small) — this is critical for cosine similarity to work. The ingestion embedding includes a metadata header ([FORTRAN SUBROUTINE 'SDR2A' in bd/sdr2a.f:1-45]) to provide context. The query embedding is just the raw query text. Despite this asymmetry, the model handles it well because the header words are semantically related to common query patterns.
Q: How do you prevent prompt injection through code content?
The code content sent to Claude is from the Pinecone-stored chunks, which were ingested from the actual NASTRAN-95 source files. Since NASTRAN-95 is a 1970s Fortran codebase, prompt injection risk is essentially zero. For a general-purpose system, I'd sanitize code content before including it in the LLM prompt, and use Claude's system prompt to instruct it to treat code blocks as data, not instructions.
Q: How does the dependency mapper find reverse dependencies (who calls X)?
Each chunk stores a dependencies list in Pinecone metadata (e.g., ["CALL:SDR2A", "COMMON:XNSTRN"]). To find who calls entity X, we search for chunks whose name matches X to find its code, then we do a second search for chunks containing CALL:X in their dependencies. Pinecone's metadata filtering makes this efficient.
Q: What are the failure modes of your reranker?
  • Synonym blindness: "matrix assembly" won't boost a chunk named "MATASS" because there's no keyword overlap. The semantic search score handles this, but the reranker can't help.
  • Common word pollution: A query about "DATA" will boost every BLOCK DATA chunk equally, providing no discriminative signal.
  • Name collision: If two routines share a name (rare in practice), both get boosted equally.
A cross-encoder would handle all these cases better, but at significant latency and cost. For this project's scale, the keyword approach is a good trade-off.
Q: How would you evaluate retrieval quality systematically?
Create a labeled evaluation set: 50 queries with manually labeled relevant chunks. Measure precision@5 (what % of top-5 results are relevant), recall@10, and MRR (mean reciprocal rank). Run this after any chunking or reranking changes. I'd also track user click-through on source cards as an implicit relevance signal.
Q: Why not fine-tune the embedding model on Fortran code?
Fine-tuning text-embedding-3-small requires OpenAI's fine-tuning API, which is more complex and expensive. For 4,040 chunks, the general-purpose model works well because: (1) Fortran keywords (SUBROUTINE, CALL, COMMON) are English words the model already understands, (2) the metadata header provides domain context, and (3) the reranker compensates for embedding imprecision. Fine-tuning would make sense at 100K+ chunks where marginal precision improvements have a larger impact.
